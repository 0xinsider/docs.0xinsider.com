| `code` | `bad_request` \| `invalid_api_key` \| `subscription_required` \| `forbidden` \| `not_found` \| `account_locked` \| `rate_limited` \| `rate_limit_unavailable` \| `internal_error` | Yes |  |

| `retry_at` | string | No | The earliest instant at which retrying is worth doing (RFC3339). Present on every retryable error (reason=pick_not_released, code=rate_limited, code=rate_limit_unavailable, reason=read_model_warming) and omitted otherwise. ALWAYS in the future. Two regimes for pick_not_released, and the value itself tells you which you are in. (1) EXACT FLOOR -- the 12-14h daily publication gap before the fixed release instant, and any day the pick is SKIPPED (a terminal day: the release loop is idle for it and nothing can publish, so retry_at is tomorrow's release slot). Nothing can publish before this instant: sleep until it and make exactly ONE request. This is the state for essentially all of the traffic. (2) BOUNDED CADENCE (~60s) -- a short interval, not a promise, returned when no floor is knowable. Two states land here: a publish is imminent (a pick is scheduled and overdue, so the release loop will publish it on its very next tick), or the release instant has passed with NO eligible candidate yet and the selector is still hunting (it re-runs every ~15 minutes until the kickoff-eligibility window closes late in the evening, and publishes within seconds of finding one -- so a pick genuinely can appear at any moment and we will not pretend to know when). You are then never more than about a minute late. Note the no-candidate case is regime 2, NOT regime 1: such a day is not over, and telling you to sleep until tomorrow would sleep you past a pick that dropped at 14:00 ET. Either way: schedule against it, do not poll. For the sleep DURATION prefer the Retry-After header, which is immune to client clock skew. |
| `reason` | `cursor_expired` \| `unknown_endpoint` \| `pick_not_released` \| `trader_not_tracked` \| `read_model_warming` | No | ADDITIVE (#7209). The specific, actionable cause behind `code`, when there is one more specific than the code itself. `code` keeps its published values, so existing clients are unaffected; new clients branch on `reason`. Omitted when the code already says everything we know. pick_not_released: no Pick of the Day is published for the current product day -- this is EXPECTED for 12-14h daily and 24h on a skipped day; do not poll, schedule against retry_at. unknown_endpoint: the PATH is not a route on this API -- read GET /api/v1, do not retry. trader_not_tracked: the wallet is real and the URL is right, but the trader is outside the HOT/WARM sync tiers -- stop asking for this wallet. cursor_expired: pagination went stale mid-walk -- re-request the first page and continue. read_model_warming: a read model is still warming; nothing is rate-limited and nothing is down -- retry after the interval. |
